## STEP 0: ブランチの継続（最初に必ず実行・スキップ絶対禁止）

**このステップを飛ばして新しいブランチで作業を始めることは禁止。**

1. `docs/CLAUDE.md` を読み、「作業ブランチ」セクションに記載されたブランチ名を確認する。
2. そのブランチが存在する場合、**そのブランチをチェックアウトして作業を継続する**。
   ```
   git fetch origin
   git checkout <ブランチ名>
   git pull origin <ブランチ名>
   ```
3. ブランチが存在しない場合（初回のみ）、mainから新しいブランチを作成する。
   ```
   git checkout main
   git pull origin main
   git checkout -b claude/nightly-$(date +%Y%m%d)
   ```
4. **新しいゲーム設計を最初から作り直すことは絶対に禁止。** docs/DESIGN.md が既に存在する場合、それが聖典であり、変更しない。

なぜこのルールがあるか:
過去にこのルールが無かったために、毎晩別のブランチ・別のゲーム設計が生成され、
8本の設計が散逸する事故が発生した。二度と繰り返さない。

## STEP 5: 実装フェーズ

Unity向け 実装時の技術的注意事項

  必ず守ること:
  - MonoBehaviourの継承はGameObject紐付きコンポーネントのみ。ロジックはPlain C#クラスで書く
  - コルーチンよりasync/awaitを優先する（UniTask推奨）
  - SerializeFieldで必要なものだけInspector露出する
  - ScriptableObjectでデータとロジックを分離する
  - シーン遷移はAddressablesまたはSceneManagerで一元管理する
  - Update()の中に重いロジックを書かない
  - null参照チェックを怠らない（?.演算子を積極的に使う）

  推奨パターン:
  - GameManager: シングルトンで全体ステート管理
  - EventSystem: ScriptableObjectベースのイベントバス
  - StateMachine: 戦闘・会話・フィールドの状態管理
  - DialogueSystem: JSONまたはScriptableObjectでデータ駆動
  - TimeManager: ゲーム内時間・カレンダー管理

Unreal Engine向け 実装時の技術的注意事項

  必ず守ること:
  - C#ロジックとBlueprintの境界を明確にする（C#: ロジック、BP: ビジュアル）
  - GameInstanceでセッションをまたいだデータを管理する
  - UMGウィジェットのコードビハインドを肥大化させない
  - SaveGameクラスで永続化データを管理する

  推奨パターン:
  - GameMode / GameState: ゲーム全体ルール管理
  - PlayerController: 入力・UIの制御
  - DataTable: キャラクター・スキルデータの管理
  - EventDispatcher: 疎結合なイベント通知

実装時の品質基準（エンジン共通）

  台詞について:
  - 説明台詞を書かない（行動・間接表現で感情を伝える）
  - 声に出して読んで、感情が動くか確認する
  - ペルソナシリーズの台詞と比較して「負けていないか」問い直す

  演出について:
  - 「ただ動く」アニメーションを作らない。感情的な意図を持たせる
  - 弱点ヒット・社会リンク上昇などの「カタルシスの瞬間」を丁寧に演出する
  - BGM・SEは仮素材でもいいので必ずつける

今夜の実装タスク候補:

  キャラクター・ストーリー系:
  - 特定キャラクターの核心的台詞の執筆・DialogueSystemへの実装
  - 社会リンクシーンのランクX〜Xの会話スクリプト
  - ストーリーイベントの追加（月次イベントの次のイベント）
  - キャラクター感情表現（表情変化・テキストエフェクト）

  戦闘系:
  - 弱点システムの実装・改善（属性相性テーブル）
  - 戦闘演出の強化（カットイン・パーティクル・SE）
  - All-out Attack相当の全体攻撃演出
  - 敵AIの行動パターン追加（StateMachine拡張）

  時間・日常系:
  - カレンダーUIの実装・改善
  - 行動選択UIの感情的演出
  - 時間経過によるイベントトリガーの実装
  - 放課後・夜の選択肢の追加

  システム系:
  - ペルソナ/能力のフュージョンシステム
  - ステータス成長システム（勇気・知識・魅力等）
  - セーブ/ロード機能
  - ScriptableObjectによるデータ整備

  演出・UI系:
  - BGM実装（Unity: AudioMixer / Unreal: Sound Cue）
  - SE実装（攻撃・弱点ヒット・社会リンク上昇など）
  - メニューUI/UXの改善
  - タイトル画面の演出強化

  品質改善系:
  - 既知バグの修正
  - コードリファクタリング
  - パフォーマンス改善（Profilerで確認）
  - 不足しているユニットテストの追加

## 付録: ペルソナシリーズから学ぶ面白さの本質

面白さの公式:
  感情移入 × システム融合 × 時間の希少性 × 演出密度

- キャラクターへの愛着がなければ、社会リンクは作業になる
- システムと感情が融合していなければ、戦闘はミニゲームになる
- 時間が有限でなければ、選択に重みが生まれない
- 演出密度が低ければ、どれだけ良い設計でも「伝わらない」

このゲームは上記公式のどこが最も弱いか、常に問い続けること。
